# GET `/api/v1/delivery-notes`

**Описание:** получение расходной накладной в формате PDF или Excel

**Запрос**

    GET /trade/hs/stock/delivery-notes

**Параметры**

| параметр                                       | допустимые значения         | описание                                                                                              | обязательный |
| ---------------------------------------------- | --------------------------- | ----------------------------------------------------------------------------------------------------- | ------------ |
| number \| deliveryNumber \| deliveryNoteNumber |                             | номер расходной накладной в 1С                                                                        | +            |
| date \| deliveryDate                           |                             | дата расходной накладной                                                                              | -            |
| format                                         | pdf \| excel \| xlsx \| xls | формат накладной (по умолчанию **pdf**)                                                               | -            |
| serials                                        | with \| without             | вариант накладной: с разделением по сериям, либо сводно по товарам без серий  (по умолчанию **with**) | -            |
**Пример запроса**
GET /delivery-notes?number=РН-000123&format=pdf&serials=without

**Успешный ответ**
Бинарный PDF-файл, не JSON.

# POST `/api/v1/delivery`

**Описание:** Метод создаёт одну расходную накладную по заказу (`ИнвойсПродажи`) на основании переданных серийных номеров (IMEI). Отгрузка всегда полная — по всем позициям заказа.

**Запрос**

    POST /trade/hs/stock/delivery

```json
{
    "orderNo": "API-0122",
    "serials": [
        5423467471235446,
        45678567867457,
        23452345674575678
    ]
}
```

```json
{
    "orderNo": "API-0122",
    "serials": [
        "5423467471235446",
        "45678567867457",
        "23452345674575678"
    ]
}
```

**Параметры**

| параметр          | описание                                                    | обязательный |
| ----------------- | ----------------------------------------------------------- | ------------ |
| orderNo \| number | входящий номер заказа (`НомерВходящий` в 1С)                | +            |
| serials           | массив серийных номеров; по одному на каждую единицу товара | +            |
**Успешный ответ**
```json
{
 "success": true,
 "error": "",
 "number": "МТ-0356879"
}
```

## Проверки (в порядке выполнения)
### 1. Авторизация (до вызова функции)

- Проверяется API-ключ запроса.
- Определяется учётная запись интеграции, от имени которой выполняется операция.
### 2. Формат запроса

| Проверка                     | Сообщение об ошибке                        |
| ---------------------------- | ------------------------------------------ |
| Указан `orderNo`             | «Укажите orderNo (входящий номер заказа).» |
| В теле есть массив `serials` | «В теле запроса нет массива serials.»      |
| `serials` — массив           | «Поле serials должно быть массивом.»       |
| После разбора массив не пуст | «Массив serials пуст.»                     |

Разбор `serials`:
- пустые значения пропускаются;
- дубликаты отбрасываются (сравнение без учёта регистра);
- числовые серии приводятся к строке.

### 3. Поиск заказа

Заказ ищется по `orderNo` = `НомерВходящий` документа `ИнвойсПродажи`.

|Проверка|Сообщение об ошибке|
|---|---|
|Заказ найден|«Заказ … не найден.»|
|Заказ принадлежит той же учётной записи API|«Заказ … принадлежит другой учетной записи API-клиента.»|
|Заказ не помечен на удаление|«Заказ … удален.»|
### 4. Состояние заказа

| Проверка                                                   | Сообщение об ошибке                                                            |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------ |
| Заказ проведён                                             | «Заказ … не проведён.»                                                         |
| Тип документа — проформа или заказ покупателя, не «Инвойс» | «Расходную накладную можно создавать только на основании проформы или заказа.» |
| По заказу ещё нет расходной накладной                      | «По заказу уже создана: … (проведена / не проведена).»                         |
> На один заказ допускается только одна расходная накладная.

### 5. Состав заказа

|Проверка|Сообщение об ошибке|
|---|---|
|В заказе есть товары|«В заказе нет товаров.»|
|Все позиции ведут серийный учёт|«Позиция «…» не ведёт серийный учёт.»|
### 6. Соответствие серий заказу

Для каждой серии из `serials`:

|Проверка|Сообщение об ошибке|
|---|---|
|Серия найдена в справочнике Партии (не помечена на удаление)|«Серия … не найдена.»|
|Товар серии входит в состав заказа|«Серия …: товар не входит в состав заказа.»|

После разбора всех серий:

|Проверка|Сообщение об ошибке|
|---|---|
|Общее число серий = общему количеству товаров в заказе|«Количество серий (N) не совпадает с количеством товаров в заказе (M).»|
|По каждой номенклатуре число серий = количеству в заказе|«По товару «…» передано серий N, в заказе M.»|

> Частичная отгрузка не поддерживается — нужно передать серии ровно на весь заказ.

### 7. Подбор партий и остатки на складе

При заполнении строк расходной накладной для каждой серии:

|Проверка|Сообщение об ошибке|
|---|---|
|На складе заказа и фирме заказа найдена ровно одна партия с остатком ≥ 1|«Не найдена партия на складе для серии ….»|
|Не найдено несколько партий одновременно|«Для серии … найдено несколько партий на складе.»|

Это единственная проверка фактического наличия товара: серия должна быть на складе заказа с ненулевым остатком.

### 8. Проведение документа

- Расходная накладная создаётся на основании заказа (контрагент, склад, валюта, цены из заказа).
- Документ записывается с проведением.
- При ошибке проведения: «Не удалось провести расходную накладную: …»

---

## Что не проверяется отдельно

- Свободные остатки по номенклатуре (как при создании заказа через `/orders`) — на этапе валидации не считаются; проверяется только наличие конкретной серии на складе.
- Статус заказа в регистре состояний (например, «готов к отгрузке») — не проверяется.
- Повторная отгрузка части заказа — не поддерживается по дизайну.


# PATCH `/api/v1/delivery`
**Описание:** Метод устанавливает статус "Собирается (заблокировано)" расходной накладной по заказу

**Запрос**

    POST /trade/hs/stock/delivery

```json
{
    "orderNo": "API-0122",
    "serials": [
        5423467471235446,
        45678567867457,
        23452345674575678
    ]
}
```

**Параметры**

| параметр          | описание                                                    | обязательный |
| ----------------- | ----------------------------------------------------------- | ------------ |
| orderNo \| number | входящий номер заказа (`НомерВходящий` в 1С)                | +            |
| employeeCode      | код сотрудника в 1С                                         | -            |
| employeePin       | пин сотрудника (используется, если не задан код сотрудника) | -            |
**Успешный ответ**
```json
{
  "success": true,
  "orderNumber": "00001234",
  "incomingNumber": "ORD-1001",
  "deliveryNumber": "00000567",
  "status": "Собирается (заблокирован)",
  "statusDate": "2026-07-13T16:30:00"
}
```
